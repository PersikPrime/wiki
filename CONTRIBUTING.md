# Участие в Superhub Wiki

Репозиторий содержит статьи, которые публикуются на
[superhub.host/wiki](https://superhub.host/wiki). Ниже описано, как добавить или
изменить статью и каким требованиям она должна соответствовать.

Часть статей пишется по техническому заданию (ТЗ), которое мы выдаём автору, а
часть — по собственной инициативе. Требования ниже распространяются на оба
случая; пункты, которые относятся только к работе по ТЗ, отмечены явно.

## Как устроен процесс

Работа над статьями ведётся через pull request'ы:

1. **Форк.** Создайте форк репозитория.
2. **Изменения.** В отдельной ветке добавьте или отредактируйте статьи. Новые
   статьи размещаются в каталоге `pages/`, при необходимости — в подходящем
   подкаталоге (например, `pages/guides/minecraft/`).
3. **Pull request.** Откройте pull request из своей ветки в ветку `main` и
   заполните шаблон PR.
4. **Ревью.** Статья проверяется по оформлению и по содержанию. Как правило,
   возникают замечания: ответьте на них и внесите правки в той же ветке — PR
   обновится автоматически.
5. **Мерж.** После устранения замечаний PR вливается в `main`.
6. **Публикация.** После мержа страница автоматически появляется на сайте.

При открытии PR автоматически запускается проверка оформления (markdown lint и
валидация `index.json`). Рекомендуется выполнить её локально заранее — см.
[Проверка перед отправкой](#проверка-перед-отправкой).

## Структура репозитория

- `pages/` — статьи в формате Markdown (`.md`).
- `index.json` — список статей, формирующий навигацию на сайте.
- `images/` — изображения, используемые в статьях.

## Добавление статьи в `index.json`

**В большинстве случаев новую статью необходимо добавить в `index.json`.** Этот
файл отвечает за отображение статьи в списке статей на сайте. Если записи в
`index.json` нет, статья откроется по прямой ссылке, но не появится в навигации.

Запись добавляется в соответствующий раздел и выглядит так:

```json
{
    "name": "Настройка иконки сервера (server-icon)",
    "path": "guides/minecraft/server-icon",
    "description": "Как подготовить и установить иконку сервера Minecraft, чтобы она отображалась в списке серверов."
}
```

- `name` — название статьи в том виде, в каком оно отображается в списке;
- `path` — путь к файлу внутри `pages/` без расширения `.md`;
- `description` — краткое описание для списка статей.

> Исключения, когда статья намеренно не добавляется в `index.json` (например,
> служебные или юридические страницы), допускаются, но обговариваются
> индивидуально. По умолчанию запись в `index.json` следует добавлять.

## Оформление файла

Каждая статья начинается с frontmatter — блока метаданных между `---`:

```markdown
---
title: 'Настройка иконки сервера (server-icon)'
description: 'Как подготовить и установить иконку сервера, чтобы она отображалась в списке серверов.'
---

Текст статьи…
```

- `title` — заголовок статьи;
- `description` — краткое описание (используется в превью и поисковыми
  системами).

### Подсказки

Для дополнительных деталей используются блоки-подсказки:

- `?> текст` — нейтральная подсказка или полезное замечание;
- `!> текст` — важное предупреждение, требующее внимания.

### Ссылки на другие статьи

Ссылки на статьи указываются относительным путём от корня сайта, например:
`[MOTD](/guides/minecraft/motd)`.

### Изображения

Каждое изображение относится к конкретной статье и используется только в её
рамках. Общих изображений, переиспользуемых между статьями, не предусмотрено: для
каждой статьи изображения хранятся отдельно.

Файлы изображений размещаются по пути, повторяющему путь статьи внутри `pages/`
(без расширения `.md`):

```
images/<путь к папке со статьёй>/<название статьи>/<название изображения>.png
```

Например, для статьи `pages/guides/configure-tcpshield.md` изображения лежат в
`images/guides/configure-tcpshield/`, а в тексте ссылка на них указывается с
ведущим слешем:

```markdown
![Список доменов](/images/guides/configure-tcpshield/domains.png)
```

## Требования к содержанию

1. Если для статьи выдано техническое задание, она должна покрывать **как
   минимум** все его пункты. Если ТЗ нет и статья пишется по собственной
   инициативе, тема и объём остаются на усмотрение автора, но статья всё равно
   должна полно раскрывать выбранную тему. В обоих случаях дополнение статьи
   собственными деталями приветствуется.
2. Больше деталей, меньше воды. Статья должна доносить мысль максимально понятно
   и при этом покрывать как можно больше вопросов читателя. Более «продвинутые»
   детали оформляются как подсказки (`?>`, `!>`).
3. Статьи индексируются поисковыми системами. Чтобы их находили, в текст следует
   добавлять синонимы и транслиты ключевых слов там, где это уместно.
   - Если упоминается возможная ошибка или строка из логов, она должна
     присутствовать в тексте **дословно**. Например, в статье про онлайн-режим
     должна встречаться фраза «Не удалось проверить имя пользователя».
4. Для статей-гайдов в конце желательно добавить раздел «Распространённые
   ошибки».
5. Стиль — дружелюбный, но профессиональный. Целевая аудитория — читатели 10–16
   лет, к которым допустимо обращаться на «ты».
6. Где можно привести пример — его **необходимо** привести.
   - Для статьи о настройке плагина приводится пример конфига в виде блока кода.
   - Для других гайдов добавляются скриншоты, показывающие ожидаемый результат.
7. К каждому изображению должно прилагаться пояснение в тексте: оно помогает
   читателю сосредоточиться на важных деталях и повышает качество обучения
   RAG-систем на статьях.
8. Оформление — в соответствии с нормами языка:
   - в русском тексте используются кавычки-«ёлочки», в английском — верхние
     кавычки `""`;
   - дефисы допускаются только внутри слов; там, где требуется длинное тире,
     ставится длинное тире «—».

## Заголовки

- Заголовки начинаются со **второго уровня** (`##`): заголовок первого уровня
  проставляется сайтом автоматически.
- Уровни заголовков уменьшаются строго на один: после `##` может следовать `##`
  или `###`, но не `####`.
- Заголовок — это заголовок, а не предложение. Если он длиннее пяти слов, вероятно,
  это должен быть обычный текст внутри абзаца. Заголовок кратко обобщает
  содержание расположенных ниже абзацев.

| Плохо | Хорошо |
|-------|--------|
| Например, эти плагины не будут работать: | Примеры нерабочих плагинов |
| Причины, по которым ваш сервер не запустится: | Распространённые ошибки |
| Мы рекомендуем перезапустить сервер после выполненных действий. | Перезапуск сервера |

## Использование языковых моделей

Использование языковых моделей допускается и приветствуется. Однако стоит
учитывать: если бы модель самостоятельно писала качественные статьи,
необходимости в авторах бы не возникало.

- Статья должна опираться в том числе на личный опыт автора — именно это придаёт
  ей ценность.
- Статьи должны выглядеть как статьи. Там, где списки не требуются, их быть не
  должно.
- Эмодзи в статьях не используются.

## Советы

Перечисленное ниже составлено по опыту ревью и помогает пройти проверку с меньшим
числом итераций.

- **Проверка русского языка.** Перед отправкой рекомендуется прогнать статью
  через любую языковую модель и проверить орфографию, пунктуацию и построение
  предложений.
- **Сокращение лишнего.** Ценится качество, а не объём. Нередко половину текста
  можно убрать без потери смысла, и читаться статья станет только лучше. Для
  проверки на «воду» удобен сервис <https://glvrd.ru/>.

## Проверка перед отправкой

Перед открытием PR рекомендуется установить зависимости и выполнить проверку
оформления — это та же проверка, что запускается в CI:

```sh
npm ci
npm run check
```

Также убедитесь, что `index.json` остаётся валидным JSON (его проверяет eslint в
CI).
